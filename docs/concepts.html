LexiGraph · Field Notes

Where this stops
being plain RAG.

Seven decisions that trip people up on first read — the two-store split, the dedupe-before-rerank trick, the verify-and-redraft loop, and the memory that flows forward. The confusing parts, drawn out.

§ 01

Storage · two stores, two jobs

One database can't do both jobs — so we run two.

Search needs meaning (“find things like payment terms”). Citation needs exact truth (“give me clause abc‑123, word for word”). A normal DB does the second; only a vector store does the first.

◆ Qdrant · vector store

Finds by meaning

  • text → [0.02,−0.5,0.9…] vectors
  • “payment terms” matches “remittance within 30 days”
  • nearest-neighbour, milliseconds over millions
  • fuzzy on purpose — where is it?
■ MongoDB · truth store

Returns exact text

  • fetch by key: parent_id → passage
  • byte-for-byte — the quote you cite
  • the parents, raw elements, jobs, outlines
  • precise on purpose — what exactly?
why not one A normal DB searching “similar meaning” would scan every row and still have no notion of similar. Qdrant finds it by meaning; Mongo proves it word-for-word. Neither can do the other's job.
§ 02

Retrieval · recall, then precision

Cast a wide, cheap net. Then score sharply.

Plain RAG takes the vector top-5 and stops — but that top-5 is only approximate. The true best passage is often ranked #8 and gets dropped. So we retrieve wide first, then rerank sharp.

the plus of qdrant RRF fusion runs on the server, fusing by rank not raw score — so you never normalise cosine against BM25 by hand. One round-trip returns the merged list.
§ 03

Retrieval · the part that confused everyone

Dedupe sits between the 50 hits and the reranker.

We do not rerank 50 raw children. Qdrant returns 50 child hits; we collapse them to distinct parents first, and only those reach Cohere.

two wins, one step Efficiency: no rerank slot wasted on a duplicate parent. Correctness: a “Net 30” clause and a conflicting “Net 60” clause are different parents — so both survive to the top 5, and the drafter surfaces the conflict instead of averaging it away.
§ 04

Retrieval · text, not vectors

The reranker reads words. No vectors cross the wire.

Stage 1 compares vectors made separately. The reranker reads the query and the passage together, as plain language — which is why it's sharper, slower, and only ever runs on the small deduped set.

◆ embedding search
  • input: vectors (numbers)
  • scores by distance between two separately-made vectors
  • cheap · runs over millions
  • approximate
■ reranker (Cohere)
  • input: query + full parent text
  • reads both together in one pass
  • expensive · runs on a handful
  • sharp
parent, not fragment Each child point stored the whole parent passage in its payload. So what gets reranked is one full parent passage per parent — never a 256-token child fragment, never anything numeric.
§ 05

Orchestration · a loop, not a line

Draft → verify → redraft-or-save → next section.

Plain RAG is one shot: retrieve, generate, done. This is a state machine (Apache Burr) with two loops living in one graph — a retry loop and a section loop — plus a durable persister that can pause for human approval and resume.

why not a while-loop The backward retry edge and the ability to pause at the human gate and resume (SQLite persister, keyed by job_id) are exactly what plain control flow does badly and a state machine does naturally.
§ 06

Safety · the hallucination firewall

D4 — no citation gets saved unless it holds.

The biggest risk in a legal tool is a fabricated citation. The evaluate node runs two checks per citation, in order — and any failure sends the section back to be redrafted.

lightweight by design This is the v1 minimum that still guarantees the safety property. The full version — re-retrieve better sources on failure (CRAG) — is deferred to Phase 2.
§ 07

Coherence · memory that flows forward

Each section is written knowing the ones before it.

In plain RAG every query is independent. Here, a growing running_summary of prior sections is fed into the next draft — so definitions stay consistent, nothing repeats, and “as set out above” actually resolves.

stand-in for index b In v1 it's the full concatenated text (fine for a short doc). At 100-page scale, this same field becomes Index B — retrieving only the relevant prior sections instead of carrying all of them. Deferred to Phase 2.
